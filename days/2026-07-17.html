1. 新增開發線：五年歷史來源回填，但不偽造歷史

  • 來源能力矩陣落地：新增 src/trustforge/historical_sources.py，把 SEC EDGAR、Alternative.me Fear & Greed、CoinGecko range、新聞 RSS、Reddit、on-chain current endpoints、HOYA ticker 分成 ready、ready_partial、credential_gated、archive_required、historical_endpoint_required 與 blocked。
  • 資料契約明確化：新增文件 docs/architecture/HERMES-FIVE-YEAR-SOURCE-BACKFILL-2026-07-16.md，要求每筆回填資料保留 provider、source、kind、published_at、實際 retrieved_at、URL、license/contract 與 content hash；retrieved_at 不可改寫成歷史時間。
  • 第一批 public history adapter：新增 scripts/fetch_public_history.py，支援 alternative-me-fng 與 sec-gov。SEC adapter 明確標記 match_scope=metadata_only，只代表官方 filing metadata lineage，不冒稱全文 Evidence。

2. 安全與誠實性：SSRF、RAG tier、stub 掃描一起收斂

  • SSRF-safe fetch 第四輪硬化：src/trustforge/ingestion/safe_fetch.py 針對 DNS pinning 後只試第一個 IP 的可用性問題，改為依序嘗試所有已驗證安全 IP；同時轉址回應不再無界讀 body，避免被巨大 redirect body 卡住或耗盡記憶體。
  • 歷史 RAG 明確降階：frontend/src/lib/endpoints.ts 要求相似歷史題目的 source_tier 必須是 historical_non_evidentiary；HermesLeftRail 也直接顯示「歷史參考 · 非本次 Evidence，不參與信任評分」。
  • 生產 stub 掃描：新增 scripts/scan_source_stubs.py 與 docs/audit/stub-allowlist.json，CI 會掃描 src/trustforge 內空函式、pass、... 與 NotImplementedError，未在 allowlist 的新 stub 會失敗。

3. Hermes upgrade control：管理 UI 看見真實覆蓋狀態

  • Upgrade Ship 顯示版本與 revision：HermesUpgradeShip.tsx 讓模組同時顯示 release version 與 revision_short，避免只看到抽象 module id。
  • 五年來源矩陣進控制室：同一頁新增 historical source matrix，把 ready、gated、archive-required、blocked 狀態展示給管理者；這延續昨天的升級治理原則：系統可以提出觀察與候選，但不能把缺資料包裝成已覆蓋。
  • 開發計畫同步：docs/plans/DEV-PLAN.md 與 HERMES-AGENT-DELIVERY-BACKLOG-2026-07-13.md 補上已處理/待處理項，讓安全修補與回填能力不只存在程式碼裡。
