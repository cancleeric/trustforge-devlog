Day 2 · 2026-07-01 — 從「能跑」到「世界第一」：研究對標與一次誠實的 revert

2026-07-01

今天不寫新功能指令，寫的是「方向」：先做功課（讀 SOTA 論文、對標世界級產品），重新定義世界第一在哪、我們差在哪；再啟動核心信任演算法深化——並記錄一次把脆弱程式碼擋在生產外的誠實 revert。

0. 重新分問：「能跑」≠「世界第一」

  • Day 1 已有可跑、可佈署、demo 有說服力的作品。但「demo 跑得動」不是「核心引擎世界級」的尺。
  • 重新分問：TrustForge 唯一能打世界第一的是信任提煉層（Trust Layer）——而它目前仍是 heuristic（詞彙重疊佐證、人工固定來源權重、regex 操縱偵測、無信心校準）。差距大且具體，不是「沒事可做」。

1. 做功課：研究 SOTA 與對標大廠

學術 SOTA（信任/可信度提煉相關）：

  • Claim verification：FEVER、AVeriTeC、RAFTS（內建對抗論證）。
  • Truth discovery（同時估來源可靠度×事實真偽）：TruthFinder、CRH、CATD。
  • 跨源佐證超越 token overlap：NLI / stance detection / claim clustering。
  • 操縱偵測：協同不真實行為（coordinated inauthentic behavior）、pump-and-dump 圖方法。
  • 信心校準：conformal prediction / abstention。

對標世界級產品：加密情報（Messari / Nansen / Arkham / Kaito / Glassnode）幾乎都是「單一維度儀表板 + 黑箱分數」；信任/溯源類（Ground News / NewsGuard / Community Notes / Perplexity）則以「可解釋、可逐層驗證」取勝，而非視覺華麗。

結論 沒有一家做「逐主張可信度 + 跨源驗證 + 反方證據 + 可解釋（非黑箱）」。這塊空白＝「情報的情報」（intelligence-about-intelligence）：別人給你分數，我們告訴你「這分數該不該信、為什麼、反方是什麼」。這是唯一能打的差異化戰場。

2. Gap 分析：核心信任引擎 vs SOTA

  • 跨源佐證：token 重疊 → 應為 NLI/語意蘊涵（gap 高）。
  • 來源信譽：人工固定權重 → 應為 truth-discovery 動態互估（gap 高）。
  • 操縱偵測：regex → 應為協同行為圖（gap 高）。
  • 判定信心：無（裸加權 + 硬門檻）→ 應為 conformal 校準 + abstain（gap 中高）。

軸線抉擇：在信任/可信度領域，世界級是靠「可解釋的深度」贏，不是視覺美術。方向定為「演算法深度 × 可解釋性」。

3. 世界第一 roadmap

  • Tier 1 護城河深化：語意佐證 → 動態來源信譽（truth-discovery）→ bridging 共識/抗操縱（Community Notes 精神）→ conformal 校準。
  • Tier 2 可解釋 UX：跨源分歧視覺化、逐項「為什麼」、來源立場/獨立性標籤。
  • Tier 3 穩健收尾：呼叫逾時保護、技術債清理。

4. W1 語意佐證：一個真 bug + 三輪對抗審 + 一次誠實 revert

真 bug：跨源佐證用「詞彙重疊」判斷，兩則語意相反但用詞相近的主張會被誤判為互相佐證、虛抬信任。例：

A: "regulatory clarity ... boost ... adoption ..."
B: "regulatory scrutiny ... boost ... investor caution ..."
# 舊邏輯：重疊高 → 判為佐證(錯);正解:語意對立,不該互相佐證

嘗試 → 對抗審 → revert：先試「反義/否定詞表」的純規則做法（零依賴、確定性）。經三輪把關：

  • Round 1（自我測試）：抓到中文單字「不」漏偵測導致假矛盾。
  • Round 2（多代理對抗式 code review）：抓到一叢 correctness 回歸——子字串誤命中（precaution 命中 caution）、否定窗只看單向、只查第一個詞、停用詞把整句清空等。
  • Round 3（獨立第二模型審查，OpenAI Codex）：抓到更深的設計漏洞——「至少兩組反義對才判矛盾」的保守規則，仍會被跨子句共現湊出假矛盾；空 token fallback 會虛構佐證。

決策 詞表規則做語意矛盾判斷本質脆弱，每補一次就冒出新的「錯殺合法佐證」——這是本模組自訂「最糟的錯誤」，比原本的 bug 更傷「信任」這個產品命脈。誠實 revert，不 ship。改用符合官方「僅限 AWS 服務提供之基礎模型」約束的做法：以 AWS Bedrock 基礎模型做逐對 NLI（entailment/contradiction/neutral）語意判斷，當成「一個特徵」餵進我方確定性評分公式（判斷結構仍由自家 pipeline 產生，守反作弊鐵則），並以結構化輸出 + 逐對快取確保可重複。

5. 教訓

  • 多層把關各有盲點：自我測試、多代理 code review、獨立第二模型，缺一不可——最深的漏洞是「獨立第二模型」抓到的。
  • 在「信任」產品裡，寧可 revert，也不讓會錯殺證據的脆弱程式碼進生產。負產出（0 行合併）但驗證了正確方向，是划算的。
  • 做功課（讀論文、對標大廠）該在「下判斷」之前，而不是之後。
