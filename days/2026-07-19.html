主要變更

  • AgentCore/strands LLM 橋接器：新增 src/trustforge/agentcore_llm_bridge.py 與 tests/test_agentcore_llm_bridge.py，在 TRUSTFORGE_AGENTCORE=1 時才 lazy import strands，保留既有 BedrockClient 介面並避免一般 CI／離線環境被新相依拖垮。
  • AgentCore 遷移計畫：新增 AGENTCORE_MIGRATION_PLAN.md，把 workshop AgentCore runtime 的目標、邊界、目錄草案、部署風險與待驗證項目寫清楚，且明確標示尚未部署、尚未建立 AWS 資源。
  • Hermes 新手體驗：新增首次使用 focus mode、三步驟 onboarding、白話結果摘要、glossary tooltip、可行動導覽說明與 beginnerExperience localStorage 記憶，降低使用者一進 dashboard 就被大量 telemetry 淹沒的風險。
  • 分析狀態誠實性：修正手動分析優先與真實 run telemetry，補上 result readiness、stage drilldown 與 live analyze fixture，讓前端更能分辨「可看上一份完整結果」與「新任務仍在跑」。
  • 營運節奏：CEO development sweep 從半小時調整成 hourly，並新增 launchd／排程測試，讓背景建議報告維持可觀測但不越權自動改 code、merge 或 deploy。
  • Demo 降級資訊收斂：隱藏不該暴露給一般 demo 使用者的內部降級細節，同時保留工程側可追蹤的模型與成本事件。

工程意義

這批 commit 的主軸不是再加一個炫功能，而是把 TrustForge 從「工程師看得懂的 Hermes 控制台」往「第一次使用也能完成可信分析」推進；同時，AgentCore bridge 以 feature flag 與 lazy import 的方式接入，讓 workshop／AgentCore 探索不污染現有 production 路徑。

對產品而言，新手教學、白話摘要與 glossary 是轉化率工程；對架構而言，AgentCore bridge 與遷移計畫則是把未來可部署到 Bedrock AgentCore 的最小切面先切出來，並把不能宣稱的地方明確寫成邊界。

驗證與測試狀態

  • 前端測試：第一次嘗試 npm test -- --runInBand 因 Vitest 不支援該 Jest 參數失敗；改跑 npm test，結果 34 files／268 tests passed。
  • 前端 build：npm run build 成功，Vite 輸出 dist/ bundle。

誠實邊界

  • 本輪沒有宣稱已部署 production，也沒有宣稱已在 workshop AWS 帳號完成真實 AgentCore deploy。
  • AgentCore bridge 的真實 Bedrock streaming 回應仍需要在具備 strands、AWS 憑證與 workshop model access 的環境驗證；本機只驗證橋接邏輯與失敗邊界。
  • 後端跑的是與本輪變更高度相關的 100 個測試，不是全倉 pytest 覆蓋率測試。
