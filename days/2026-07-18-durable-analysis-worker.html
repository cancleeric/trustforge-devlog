Day 19 · Durable Analysis Worker 與手動分析佇列收斂

TrustForge 主線自上一篇 devlog 之後新增 33 個 commit，版本推進到 v0.16.12；核心是把 production 分析流程改成可持久化 worker、手動分析高優先佇列、Hermes 低優先排程與 runtime 控制面，並補齊前端狀態、API 文件、部署腳本與競賽收斂文件。

主要變更

  • Production 分析 worker 持久化：最新 HEAD fix: run durable analysis worker in production 把 production 分析改成由 durable worker 消化工作，避免手動分析只停留在 ephemeral request path。
  • 手動分析高優先佇列：feat: prioritize durable manual analysis jobs 與 v0.16.10–v0.16.11 的 changelog 顯示，手動 run 與 Hermes 自動排程已切開；手動工作可持久化、優先處理，並回報目前節點、佇列位置、錯誤與完成報告。
  • Hermes 排程降為背景節奏：Hermes 每 30 分鐘建立低優先工作，worker 仍會優先消化手動工作；同時新增 CEO sweep 與半小時排程安裝／解除腳本，讓自動開發節奏可被部署層管理。
  • Runtime 控制面：新增 src/trustforge/runtime_control.py、scripts/trustforge_control.sh 與對應測試，讓 production 可用更明確的方式控制 Hermes／分析服務狀態。
  • 前端狀態收斂：AnalyzePage、HermesDashboard、HistoryPage、StatusStates 與 API client/type/validator 有同步更新，對應新的佇列與分析生命週期。
  • 文件與競賽收斂：新增 Kiro steering、競賽 build plan、creative strategy、設計稿與 finale two-week execution plan，將 TrustForge 的「信任層／透明分析」敘事與交付節奏具體化。

工程意義

這批變更把 TrustForge 從「按下手動分析後等待同步流程」推向「可持久化、可排隊、可觀測、可由 runtime 控制」的服務模型。手動分析與 Hermes 排程分流後，使用者操作不再被背景節奏綁住；而 worker、佇列位置、錯誤與完成報告的可見性，能支撐更可靠的 production demo 與後續營運。

部署與文件層也同步往競賽收斂：版本已到 v0.16.12，README、OpenAPI、EC2 deploy、nginx、launchd/EventBridge 腳本都被更新，代表這不是單一前端修補，而是跨 API、worker、部署與敘事的一次 release hardening。
