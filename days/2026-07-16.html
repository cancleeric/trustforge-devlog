Day 17 · Hermes 持續智慧 + 升級治理控制平面 + 五年歷史回填

2026-07-16

分類：Hermes ｜ 標籤：#hermes #continuous-intelligence #upgrade-governance #sqlite #rag #historical-backfill

Hermes 從「能跑的 demo」跨進「會自己持續思考、但凡事先問人」的治理型系統。今天 17 個 commit 串起兩條主軸：讓分析變成持續、可檢索、可追線的智力流水線，以及把「系統升級自己」這件事收進一層層需要人類批准的閘門。

1. Hermes 持續智慧（Continuous Intelligence）

  • 五階段持續分析矩陣：所有幣／模式／活躍問題跑 snapshot-isolated 的持續分析（analysis_flow.py，664 行新邏輯），執行鎖定在不可變快照上，UI 導航保持唯讀並在交換時保留最後完整快照。
  • SQLite 取代 JSON 快取／帳本：本地快取與成本帳本從 JSON 檔遷移到 SQLite，附可重複執行的遷移工具（migrate_json_cache_to_sqlite.py / migrate_json_ledger_to_sqlite.py），快取層重寫 192 行。
  • 問答 RAG + 對話 lineage：加入 SQLite 問題 RAG、使用者／Hermes 對話紀錄、以及 run／snapshot 的來源 lineage；/api/analysis-question-context 實測回傳 SQLite 檢索、五筆匹配、對話歷史、job_id 與 snapshot_id。
  • 真實遙測下鑽：統一幣種選擇、暴露真實管線遙測與下鑽；HermesDashboard 大改（390 行變動），修正遙測輪詢競態（serialize polling），左右欄職責與右欄空間利用率重新規劃。
  • 本地守護進程：新增 analysis-flow / local-refresh / local-web 三個 launchd plist，讓持續智慧在本地背景常駐；並有 sync_cloud_cache_to_local.py 把雲端快取同步回本地。

2. 升級治理控制平面（Upgrade Governance）

  • Flagship 升級控制平面：upgrade_control.py 新增 89 行控制平面，註冊完整的「外部升級介面」（outer upgrade surface），把升級建模為系統拓撲而非零散腳本。
  • Data-driven LLM 升級審查清迴圈：upgrade_review.py 讓 LLM 基於歷史管線測量結果，提出需人類批准的「外部框架改進提案」；前端 UpgradeShip 顯示提案與釋出版本號。
  • Durable 人類升級閘門：升級候選人持久化進 upgrade queue（含本地前端 plist），重啟後重試狀態與 DLQ 不丟失；新增 durable human upgrade gate，寫入端點限速、待辦受容量約束。
  • Sandbox 啟用治理：sandbox 啟用納入治理（run_skill_sandbox.py + upgrade_queue.py 81 行），沙盒重播（replay）不碰生產。

3. 可審計歷史來源回填

  • SEC 歷史元資料配接器：historical_sources.py 新增 SEC historical metadata adapter，fetch_public_history.py 支援回填。
  • 五年可審計來源回填：建立可審計的五年歷史來源回填流程，回填的歷史結論僅作參考、絕不進入當前 Trust Evidence——誠實邊界再次畫清。

安全與誠實底線

  • 改進一律「只提案」：sandbox/replay + 人類批准才放行，禁止自動生產變更；寫入端點限速、待辦容量有界。
  • 檢索到的歷史結論標記為非證據性，不污染當前信任證據。
